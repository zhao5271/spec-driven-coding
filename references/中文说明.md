# Spec 驱动编码 Skill 中文说明

这是一份便于人工阅读的中文说明版。真正用于 skill 触发和执行的主文件仍然是 `SKILL.md`。

如果后续 `SKILL.md` 的流程、脚本入口、模板结构有改动，这份中文说明也要同步更新，避免长期偏离。

## 这个 Skill 是做什么的

这个 skill 用来给任意代码仓库落地一套轻量的 Spec 驱动编码流程。

它也是这套流程里的主流程 skill。即使仓库里同时安装了 `superpowers` 一类流程型 skill，也应该由它来管理变更包结构、`spec.md`、`tasks.md` 和 Reverse Sync。

它不预设你必须使用 Go、Python、Node.js、Java 或某个固定前后端框架。它的核心目标是：

- 先把项目规则写进文件，而不是只留在对话里
- 先把需求写成 `spec.md`
- 再把工作拆成 `tasks.md`
- 最后按任务执行，并在实现偏离方案时做 Reverse Sync

适合的场景：

- 给一个现有仓库初始化 `code_copilot/`
- 为某个新需求或 bug 创建 `spec.md / tasks.md / log.md`
- 在真正编码前先把现状、边界、风险、验证方式写清楚
- 在多层改动里统一前端、API、数据库、异步任务的执行节奏
- 需要根据工作切片，把任务路由到更专门的 companion skill

## 最小执行包长什么样

在目标仓库里会创建：

```text
code_copilot/
├── agents/
├── rules/
├── knowledge/
└── changes/
    └── templates/
```

其中核心作用是：

- `rules/`：放稳定规则，比如项目上下文、代码风格、安全要求、业务边界、接口契约
- `knowledge/`：放长期有效的知识沉淀，比如历史坑点、状态规则、缓存规则、集成注意事项
- `changes/templates/`：放每次需求变更会用到的模板和任务样例
- `agents/`：放给 AI 协作时使用的固定执行提示

## 什么时候用哪个脚本

### 1. 初始化仓库

当你要给一个仓库接入这套工作流时，运行：

```bash
python3 scripts/scaffold_package.py \
  --target /path/to/repo \
  --project-name my-app \
  --runtime "python" \
  --backend "fastapi" \
  --frontend "react + vite" \
  --database "postgres" \
  --cache "redis"
```

可选参数还有：

- `--identifier`
  用来记录模块名、包名、workspace 名或仓库标识
- `--async-stack`
  用来记录队列、worker、流处理、任务系统
- `--test-command`
  用来记录这个仓库默认的验证命令

脚本会生成这些核心文件：

- `rules/project-context.md`
- `rules/coding-style.md`
- `rules/security.md`
- `rules/domain-rules.md`
- `rules/api-contracts.md`
- `rules/response-pattern.md`
- `rules/recommended-skills.md`
- `rules/skill-decision-table.md`
- `knowledge/index.md`
- `agents/copilot-prompt.md`
- `agents/execution-sop.md`
- `changes/templates/spec.md`
- `changes/templates/tasks.md`
- `changes/templates/log.md`

还会额外生成几类高频任务样例和变更样例：

- `changes/templates/task-examples/api-endpoint-tasks.md`
- `changes/templates/task-examples/ui-flow-tasks.md`
- `changes/templates/task-examples/background-job-tasks.md`
- `changes/templates/task-examples/data-migration-tasks.md`
- `changes/templates/examples/api-change.md`
- `changes/templates/examples/ui-change.md`
- `changes/templates/examples/async-job-change.md`
- `changes/templates/examples/data-migration-change.md`

这些不是强制格式，而是帮助 AI 和人工更快进入“可执行 spec”状态的参考材料。

### 2. 创建一次具体需求的变更包

当你要开始一个真实需求或 bug 修复时，先创建变更包：

```bash
python3 scripts/create_change.py \
  --target /path/to/repo \
  --name add-bulk-import \
  --title "Add bulk import flow"
```

它会生成：

- `code_copilot/changes/add-bulk-import/spec.md`
- `code_copilot/changes/add-bulk-import/tasks.md`
- `code_copilot/changes/add-bulk-import/log.md`

建议顺序：

1. 先写 `spec.md`
2. 再拆 `tasks.md`
3. 用户确认后再正式编码

## spec.md 应该写什么

最小结构建议是：

1. 背景与目标
2. 当前代码现状
3. 功能变更点
4. API / 数据 / 集成变更
5. 风险与评审点
6. 测试与验证方案
7. 未决问题

其中“当前代码现状”要尽量写真实文件路径、入口点、函数名、服务路径、页面路径，不要让模型凭空脑补架构。

如果变更涉及下面这些内容，建议对照 `references/spec-checklists.md`：

- 公共 API 或前后端契约
- 数据库结构或迁移
- 缓存、队列、异步 worker
- 页面、表单、状态流
- 认证、鉴权、敏感数据
- 配置、发布、回滚、运维观察点

## tasks.md 应该怎么拆

任务要“小而可验证”。

推荐原则：

- 一个任务只表达一个清晰意图
- 一个任务尽量只覆盖 3 到 5 个文件
- 每个任务都要写验证方式
- 文件名、函数名、入口点尽量写具体

好的任务示例：

- 增加请求 schema 与边界校验
- 实现导入流程的 service 分支
- 增加状态索引与迁移脚本
- 更新客户端请求映射与结果渲染

不好的任务示例：

- 完成后端
- 处理所有 worker 逻辑
- 改完前端

## 这套流程最重要的三条规则

### No Spec, No Code

没有 Spec，不开始写代码。

### Spec is Truth

执行期间，以 `spec.md` 作为当前方案真相来源。

### Reverse Sync

如果代码实现和原方案不一致：

1. 先停下来
2. 先更新 `spec.md`
3. 如果执行步骤变了，再更新 `tasks.md`
4. 再继续写代码

不要让代码偷偷偏离方案。

## 这个 Skill 特别强调什么

### 1. 不强推固定技术栈

这个 skill 是通用版，不应该把某个语言或框架的习惯强行套到所有仓库上。

你需要尽早在 `rules/project-context.md` 里写清：

- 语言或运行时
- 包结构或 monorepo 结构
- 后端入口、前端入口、任务入口
- 测试、构建、迁移、部署命令
- 配置、密钥、发布边界

### 2. 规则、知识、变更记录分开存

- `rules/` 放稳定规则
- `knowledge/` 放长期复用知识
- 单次需求中的临时发现、权衡、偏差说明，放在每个 change 的 `log.md`

### 3. 先统一规划，再按领域路由

这个 skill 适合先统一管理 spec 和任务，再根据工作切片调用更专门的 skill：

- 前端页面、组件、样式、交互：
  `frontend-design`
- API 设计、契约、分页、错误返回：
  `api-design-principles`
- PostgreSQL 表设计、索引、约束：
  `postgresql-table-design`

## 和 superpowers 怎么配合

如果你同时安装了 `superpowers`，推荐把它当作执行增强包，而不是第二套主流程。

角色分工建议固定成这样：

- `spec-driven-coding` 管主流程
- `superpowers` 只管执行方法

具体来说：

- `spec-driven-coding` 负责：
  - 创建变更包
  - 维护 `spec.md`
  - 拆分 `tasks.md`
  - 在实现偏离方案时执行 Reverse Sync
- `superpowers` 执行型子 skill 负责：
  - 怎么写测试
  - 怎么调试
  - 怎么做收尾验证
  - 怎么发起代码复核

### 推荐组合

#### 新功能 / 重构

- 主 skill：`spec-driven-coding`
- 执行辅助：`test-driven-development`
- 收尾验证：`verification-before-completion`

推荐顺序：

1. 先创建 change package
2. 先写清 `spec.md`
3. 再拆 `tasks.md`
4. 按 task 执行，用 TDD 落代码
5. 完成前做最终验证

#### 修 bug / 排查异常

- 主 skill：`spec-driven-coding`
- 排查辅助：`systematic-debugging`
- 收尾验证：`verification-before-completion`

推荐顺序：

1. 先把现象、影响范围、当前假设写进 `spec.md`
2. 把调查任务和修复任务写进 `tasks.md`
3. 按系统化调试方式定位根因
4. 如果根因和原判断不同，先更新 spec 和 tasks
5. 修完后做验证

#### 大改动 / 高风险变更

- 主 skill：`spec-driven-coding`
- 复核辅助：`requesting-code-review`

推荐顺序：

1. 保持 `spec.md` 和 `tasks.md` 始终最新
2. 完成当前变更目标
3. 发起代码复核
4. 如果复核意见改变了方案边界，先更新 spec 再继续

### 不要这样用

下面这些做法会让两套流程打架，应该避免：

- 不要把 `using-superpowers` 当顶层规则
- 不要同时维护两套主计划：
  - `code_copilot/changes/...`
  - `docs/superpowers/plans/...`
- 不要让 `writing-plans` 取代 `code_copilot/changes/<change-name>/tasks.md`

### 冲突时怎么裁决

如果 `spec-driven-coding` 和某个 `superpowers` skill 的说法冲突，按下面的优先级处理：

1. `code_copilot/changes/<change-name>/spec.md` 是唯一方案真源
2. `code_copilot/changes/<change-name>/tasks.md` 是唯一任务清单
3. `superpowers` 只提供执行方法，不接管规划
4. 一旦实现偏离方案，先做 Reverse Sync，再继续编码

可以把这几条规则压缩成一句：

- No Spec, No Code
- Spec is Truth
- One change, one planning system
- `superpowers` 只增强执行，不改主流程

## 一句话理解

这个 skill 的目标，不是单纯“帮你写代码”，而是让任意技术栈下的 AI 协作开发也能先有规则、再有方案、再有任务、最后才进入实现。
